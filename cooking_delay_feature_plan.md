# 조리 지연 자동화 기능 개발 정리

## 목적

기존 자동 수락 봇과 별개로, 쿠팡이츠 POS에서 조리 지연 시간을 추가하는 기능을 신설한다.

기존 자동 수락 봇은 주문 팝업을 감지하고 수락하는 흐름을 유지한다. 새 기능은 자동 수락 루프에 섞지 않고, 별도의 실행 흐름 또는 GUI 버튼으로 동작하는 독립 기능으로 만든다.

## 원하는 동작

1. 고정되어 있지 않은 네이티브 POS 메인창을 찾는다.
2. POS 메인창이 닫혀 있거나 보이지 않는 경우, POS를 열 수 있는 팝업 또는 알림을 찾아 메인창을 연다.
3. POS 메인창에서 `조리 지연` 버튼을 누른다.
4. 조리 지연 화면 또는 팝업에서 `시간 추가` 버튼을 누른다.
5. 성공 또는 실패 상태를 사용자에게 알려준다.

## 기존 자동 수락 봇과의 관계

이 기능은 기존 자동 수락 봇과 분리한다.

- 기존 자동 수락 기능은 그대로 유지한다.
- 새 기능은 별도 모듈로 만든다. 예: `delay_bot.py`
- 공통으로 쓸 수 있는 기능만 기존 `bot.py`에서 재사용한다.
  - 화면 캡처
  - 이미지 매칭
  - 클릭 함수
  - OS/Retina 좌표 처리
- 자동 수락 카운트, 주문 팝업 감지 루프, 수락 실패 감지 로직과는 연결하지 않는다.

## 주요 리스크

### POS 창 위치와 크기가 가변적임

사용자가 POS 메인창을 이동하거나 크기를 조절할 수 있으므로, 특정 기준점에서의 상대좌표 클릭은 위험하다.

따라서 새 기능은 상대좌표 기반이 아니라 다음 방식으로 구현하는 것이 안전하다.

1. UI Automation으로 버튼 자체를 찾는다.
2. 실패하면 이미지 매칭으로 버튼 자체를 찾는다.
3. 상대좌표 클릭은 사용하지 않거나 최후의 fallback으로만 둔다.

### 다른 창이 POS를 가릴 수 있음

다른 창이 POS 메인창을 가리고 있으면 이미지 매칭 기반 클릭이 실패하거나 잘못된 곳을 클릭할 수 있다.

따라서 클릭 전에는 다음 검증이 필요하다.

1. POS 창을 찾는다.
2. POS 창을 복원하고 전면화한다.
3. 실제 화면에 POS 메인창이 보이는지 확인한다.
4. POS 창 영역 안에서만 버튼을 탐색한다.
5. 버튼을 찾지 못하면 추측 클릭하지 않고 실패 처리한다.

## 가장 먼저 확인해야 할 것

가장 먼저 확인할 것은 POS 앱의 버튼들이 Windows UI Automation 또는 접근성 트리에 잡히는지 여부다.

확인 대상:

- POS 메인창의 창 제목
- POS 메인창의 프로세스명
- `조리 지연` 버튼이 UI Automation에서 보이는지
- `시간 추가` 버튼이 UI Automation에서 보이는지
- 각 버튼이 실제 클릭 가능한 컨트롤인지

결과에 따른 구현 방향:

- 버튼이 UI Automation으로 잡힘
  - `pywinauto` 또는 UI Automation 기반으로 구현한다.
  - 창 크기와 위치 변화에 가장 강하다.
- 텍스트는 잡히지만 버튼 컨트롤은 안 잡힘
  - 텍스트 위치 또는 부모 컨트롤 클릭 가능성을 확인한다.
- 아무것도 잡히지 않음
  - 이미지 매칭 기반으로 구현한다.

## 추천 구현 순서

### 1단계: 진단 스크립트 작성

Windows POS 환경에서 UI Automation 정보를 확인하는 작은 진단 스크립트를 만든다.

목표:

- 현재 열린 창 목록 출력
- POS 후보 창 제목과 프로세스명 출력
- POS 창 하위 컨트롤 목록 출력
- `조리 지연`, `시간 추가` 텍스트가 잡히는지 확인

예상 파일:

- `inspect_pos_ui.py`

### 2단계: 별도 기능 모듈 추가

자동 수락 봇과 분리된 조리 지연 모듈을 만든다.

예상 파일:

- `delay_bot.py`

예상 함수:

```python
def find_pos_window():
    """POS 메인창을 찾는다."""


def ensure_pos_main_open():
    """POS 메인창을 찾고, 없으면 팝업 또는 알림을 통해 연다."""


def click_cooking_delay():
    """조리 지연 버튼을 찾아 클릭한다."""


def click_add_time():
    """시간 추가 버튼을 찾아 클릭한다."""


def run_once():
    """조리 지연 시간 추가를 1회 실행한다."""
```

### 3단계: 클릭 백엔드 추상화

현재는 `click_at()`에서 `pyautogui` 또는 macOS `osascript` 클릭을 사용한다.

향후 커서 이동 없는 OS API 클릭으로 바꿀 예정이므로, 클릭 함수는 기존 코드와 새 기능이 함께 사용할 수 있게 추상화한다.

권장 방향:

```python
def click_at(x, y):
    if platform.system() == "Windows":
        os_click_windows(x, y)
    elif platform.system() == "Darwin":
        os_click_mac(x, y)
    else:
        pyautogui.click(x, y)
```

주의:

- 화면 좌표 기반 OS API 클릭은 버튼이 실제로 화면에 보여야 안정적이다.
- 가려진 창까지 클릭하려면 UI Automation 또는 창 핸들 기반 메시지 클릭을 별도로 검증해야 한다.
- POS 앱 종류에 따라 `PostMessage` 방식은 동작하지 않을 수 있다.

### 4단계: 이미지 매칭 fallback 준비

UI Automation이 불가능할 경우를 대비해 템플릿 이미지를 준비한다.

예상 템플릿:

- `templates/delay_pos_popup.png`
- `templates/pos_main_anchor.png`
- `templates/cooking_delay.png`
- `templates/add_time.png`

이미지 매칭 시 주의:

- 화면 전체가 아니라 POS 창 영역 안에서만 찾는다.
- 매칭 정확도 기준을 둔다.
- 버튼이 여러 개 잡히면 가장 신뢰도 높은 후보만 사용하거나 실패 처리한다.
- 버튼을 못 찾으면 추측 좌표 클릭을 하지 않는다.

## 권장 실행 흐름

```text
조리 지연 실행
↓
POS 메인창 찾기
↓
없으면 POS 열기 팝업/알림 찾기
↓
POS 메인창 복원 및 전면화
↓
POS 화면이 실제로 보이는지 검증
↓
조리 지연 버튼 찾기
  1순위: UI Automation
  2순위: 이미지 매칭
↓
조리 지연 버튼 클릭
↓
조리 지연 화면/팝업 표시 확인
↓
시간 추가 버튼 찾기
  1순위: UI Automation
  2순위: 이미지 매칭
↓
시간 추가 버튼 클릭
↓
완료 상태 확인
```

## 실패 처리 원칙

이 기능은 POS 화면을 직접 조작하므로 실패 시 안전하게 멈춰야 한다.

다음 상황에서는 클릭하지 않고 실패 메시지를 반환한다.

- POS 창을 찾지 못함
- POS 창 전면화에 실패함
- POS 화면이 실제로 보이는지 확인하지 못함
- `조리 지연` 버튼을 찾지 못함
- 조리 지연 화면 또는 팝업이 뜬 것을 확인하지 못함
- `시간 추가` 버튼을 찾지 못함
- 클릭 후 상태 변화가 확인되지 않음

## GUI 적용 방향

기존 GUI에는 자동 수락과 별도의 버튼을 추가한다.

예:

- `조리 지연 +시간`

동작 방식:

- 사용자가 버튼을 누르면 `delay_bot.run_once()`를 실행한다.
- 실행 중에는 버튼을 잠시 비활성화한다.
- 성공/실패 메시지를 GUI 상태 영역에 표시한다.
- 기존 자동 수락 루프가 실행 중이어도 이 기능과 충돌하지 않게 상태 관리를 분리한다.

## 결론

기능적으로 구현 가능하다.

다만 POS 창 크기와 위치가 가변적이고, 다른 창이 POS를 가릴 수 있으므로 상대좌표 기반 구현은 피한다. 먼저 UI Automation으로 버튼이 잡히는지 확인하고, 가능하면 그 방식으로 구현한다. 불가능할 경우에는 POS 창 영역 안에서 이미지 매칭으로 버튼을 찾는 fallback을 사용한다.

가장 먼저 할 일은 Windows POS 환경에서 UI Automation 덤프를 떠서 `조리 지연`과 `시간 추가` 버튼이 잡히는지 확인하는 것이다.
